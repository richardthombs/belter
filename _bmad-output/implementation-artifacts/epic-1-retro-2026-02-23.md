## What Went Well

### 1. Naming conventions locked in on day one
Story 1.1's `UseSnakeCaseNamingConvention()`, JSON `camelCase`, and C# naming rules were established and held throughout all 8 stories without a single violation. Zero convention drift across 96 tests and all EF Core migrations.

### 2. Dev Notes as institutional memory
Debug discoveries were explicitly cross-referenced in subsequent stories. The `IShardClient`/`ShardClient` typed client pattern from Story 1.3 was reused without friction in Story 1.5. The `ProviderName` guard from Story 1.3 was referenced in Story 1.4. This forward-propagation prevented repeated mistakes.

### 3. Code review as quality gate — 100% completion
Every story received a full code review before being marked done. No shortcuts taken. This discipline paid dividends.

### 4. Clear velocity improvement over the epic
Stories 1.5–1.7 were noticeably cleaner implementations than 1.1–1.3. The compound learning effect was measurable — Story 1.6 had zero debug log entries. Story 1.5 had one.

### 5. Architecture decisions kept consistent
The rendering layer order (Background → World → Effects → UI), the SignalR/REST split, and the `InputEvent` abstraction all held exactly as designed. No architectural u-turns.

## Challenges

### 1. EF InMemory provider divergence (appeared in 2 stories)
The InMemory provider does not support real transactions, does not support `IsRelational()`, and requires a special `UseInternalServiceProvider` override for integration tests. This pattern was documented in Story 1.2 and 1.3 debug logs but needs a permanent home in `project-context.md`.

### 2. Code review always found something
8/8 stories required fixes post-review. Stories 1.3 and 1.4 had HIGH-severity findings. This is healthy — it validates the process — but signals that initial implementations consistently omit something at the MEDIUM level. Common misses: resource disposal, error handling for network failures, accessibility attributes.

### 3. PixiJS v8 API gaps in training data
The dev agent's training data contains v7 patterns. Key v8 differences (`Application.init()` is async, `cacheAsTexture()` replaces `cacheAsBitmap`) required active documentation in story files. Epic 2 will have more PixiJS work — these patterns need to be in `project-context.md`.

### 4. Production deployment carried as unresolved debt throughout the epic
`deploy.yml` has been a `TODO` placeholder since Story 1.1. Story 4.0 addresses this, but the gap was carried the full epic. Epic 2 can proceed on docker-compose without risk, but Story 4.0 must be completed before Epic 4 begins.

### 5. Story 1.6 required two code review passes
The first pass missed a HIGH finding: `e.preventDefault()` blocking all concurrent finger scrolling on touch devices. Clean initial implementation ≠ complete implementation. Complex UX interactions warrant extra scrutiny.

---

## Key Insights

1. **Stories with explicit Dev Notes + architecture references are cheaper to implement and review.** The best stories (1.5, 1.6) had dense cross-references. The roughest stories (1.1, 1.3) had the most debug surprises.

2. **Latent bugs surface in later stories.** Story 1.7's main work was actually fixing a bug in `SpawnController` introduced in Story 1.3 — returning `(0, 0)` instead of the player's saved position. The spawn story revealed the bug. Story framing doesn't always surface the real risk.

3. **The InMemory EF pattern is boilerplate-prone.** Every future story that touches database code will potentially hit the same InMemory-vs-Postgres behaviour divergence. This must be documented centrally.
